## §1 — Session startup

1. Read **CLAUDE.md** for the project
2. Read **docs/96-project-structure.md** (or project equivalent)
3. Read **docs/97-skill-orchestration.md** (or project equivalent)
4. Read **AGENTS.md** at the root if present (DS contract generated by Lyse)
5. Activate **plan mode** — all tasks go through a plan before implementation

---

## §2 — Plan and feasibility scoring

### Required template

Every plan must follow this exact format:

```md
## Plan — [Task name]

### Context
Why this task exists — the problem or need it addresses.

### Steps
- [ ] Step 1 — description [Feasibility: XX%] → Skill: X
- [ ] Step 2 — description [Feasibility: XX%] → Skill: X

### Verification
How to test that the task is complete (run, MCP, test, browser).
```

Each step is annotated with a score:

```
- [ ] Step X — description [Feasibility: 87%] → Skill: design-taste-frontend
```

**If score < 90%:**
Re-run the routine on that specific point:
1. Identify the exact blocker (ambiguity, missing dependency, undecided decision)
2. Propose a rephrasing or alternative approach
3. Re-evaluate → new score
4. Repeat until ≥ 90% **or** identify that the blocker is out of scope

If out of scope → escalate to Maxime, do not improvise.

**Verification command:** `/plan-check` — checks that the plan follows this format.

## §3 — Skill routing

**Convention:** every protocol step that triggers an action (code, design, content) annotates the applicable skill as `→ Skill: X`. If no skill is required: `→ Skill: none`.

| Task | Skill |
|---|---|
| Generic code (utils, data, config) | `ponytail` |
| Front — full page, section, layout | `ui-ux-pro-max` |
| Front — isolated component | `frontend-component-build` |
| Tokens / palette / typography | `design-system` + `design-standards` |
| Content (texts, case studies) | `content-and-copy` |
| Animation / Motion / Micro-interactions | `gpt-tasteskill` |
| Accessibility audit | `design-standards` |

### Unknown or uninstalled skill
1. Check https://github.com/rampstackco/claude-skills
2. Propose the most fitting skill to Maxime with justification
3. Wait for confirmation + installation before acting
4. In remote session: ask Maxime for the skill doc if not available locally

### rampstackco fallback
If no skill matches → explain why, propose the manual approach, ask for validation.

## §4 — End of task: documentation

After any task that creates or modifies a component, file, token, or decision:

→ Skill: `ponytail` (for all code written or modified in this task)

1. **Component created or modified** → create/update the co-located `.md` (template in §8)
2. **Components index** → update `docs/98-components-index.md`
3. **Potentially impacted `.md` files** → propose updates to ALL affected files,
   not just `/docs`:
   - `docs/96-project-structure.md` (structure, conventions)
   - `docs/97-skill-orchestration.md` (routing)
   - `docs/98-components-index.md` (index)
   - `CLAUDE.md` (project rules)
   - `AGENTS.md` (DS contract — regenerate via `lyse agents > AGENTS.md`)
   - `README.md` if present

## §5 — End of plan: Lyse audit

After each executed plan (not after each file):

```bash
npx lyse audit
```

- Report the **Health Score** in the session thread
- If regression vs previous score → identify the faulty component or token before closing
- If improvement → note the tokens/components that improved
- If tokens or components changed: **propose a diff** of what changed in AGENTS.md, never overwrite blindly.
  - Run `npx lyse agents` to a temp file, compare with the current version, and submit only the relevant changes to Maxime for validation before overwriting.

## §6 — Figma → code workflow

### Rule 0 — MCP first

Always use the Figma MCP (`mcp__Figma__*`) to analyze a design.
A screenshot yields a significantly lower-quality analysis.
**Never implement based solely on a screenshot.**

### Step 1 — Complexity assessment [before analysis]

Evaluate before starting:

| Level | Criteria | Action |
|---|---|---|
| **Simple** | < 5 components, 1 state, no multiple breakpoints | Direct analysis |
| **Moderate** | 5–15 components, a few variants, 2 breakpoints | Structured analysis plan |
| **Complex** | > 15 components, interactions, custom tokens, multiple breakpoints | **Flag to Maxime**, propose decomposition by blocks |

If complex: do not analyze in one pass → break into subtasks, validate with Maxime before proceeding.

### Step 2 — MCP analysis sequence

```
1. mcp__Figma__get_metadata       → general context of the Figma file
2. mcp__Figma__get_design_context → detailed structure (nodes, tokens, layout, variants)
3. mcp__Figma__get_screenshot     → supplement only if a visual detail is ambiguous
```

### Step 3 — Analysis report format

```md
## Figma Analysis — [Component/screen name]

### Complexity: Simple / Moderate / Complex

### Identified components
- [ComponentName] — role

### Tokens used
- Color: `#hex` → `--color-X` (DS token)
- Typography: weight/size → hierarchy token
- Spacing: value → Tailwind class

### Design system alignment [XX%]
- ✅ Already covered (codebase / AGENTS.md)
- ⚠️ Missing (new component, new token)
- ❌ Divergences from decisions in 93/95/96

### Implementation plan
- [ ] Step 1 [Feasibility: XX%]
- [ ] Step 2 [Feasibility: XX%]
```

### Step 4 — Alignment score

```
Alignment (%) = elements covered by DS / total identified elements × 100
```

- **≥ 70%** → implement
- **< 70%** → alert Maxime before implementing (risk of DS drift)

### Figma skills

| Skill | When |
|---|---|
| `/figma-use` | **REQUIRED** before any call to `use_figma` |
| `/figma-code-connect` | Map Figma components ↔ codebase components |
| `/figma-generate-design` | Push code to Figma |

## §9 — Task implementation order

Every task follows this order. Do not skip a step.

1. **Structure** — HTML/JSX, layout, components, props, Server vs Client, accessibility
   → Skill: `frontend-component-build` (isolated component) or `ui-ux-pro-max` (page/section) — see §3
2. **Content** — texts, typography tokens, color tokens, data
   → Skill: `content-and-copy`
3. **Animation / Motion / Micro-interactions** — GSAP, CSS transitions, hover/focus states
   → Skill: `gpt-tasteskill`
   - If not specified in the request → propose the planned animations, wait for validation
   - Always implement `prefers-reduced-motion`



```md
# ComponentName

## Role
One sentence — what this component does, not how.

## Props
| Prop | Type | Default | Description |
|---|---|---|---|

## Lyse tokens used
- `--color-X` → Tailwind class `bg-X`

## Variants / states
List of variants or interactive states.

## Design decisions
Why this choice — hidden constraint, trade-off, workaround.
Do not document what the code already says.
```
